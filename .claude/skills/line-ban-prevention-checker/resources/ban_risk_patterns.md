# リスクパターンと対策

## 1. 本質理解：なぜ凍結・不達が起きるのか

### 1.1 プラットフォームの立場
LINE・Gmailなどのプラットフォームは「ユーザー保護」を最優先としている。

**プラットフォームの目的：**
- ユーザーが安心・安全にサービスを利用できること
- 詐欺・スパムからユーザーを守ること
- ユーザー体験を損なう行為を排除すること

**LINE公式ガイドラインより：**
「情報の受け手（ユーザー）がどう思うか」「ユーザーが不快に感じることはないか」 「すべてのユーザーが安心・安全にLINEを利用できているか」という点を非常に重視

### 1.2 凍結・不達の根本原因
「詐欺・スパム業者と同じパターン」を取ると排除される

**判定の本質：**
- 単語単体ではなく「パターン」で判断される
- コンテンツ＋行動＋リスト品質の複合評価
- 「詐欺業者がやりそうなこと」は全てリスク
- 通報が最も影響力が大きい

### 1.3 推定される凍結判定メカニズム
**[第1層：自動フィルター]**
- 禁止ワード検知
- URL頻度異常検知
- 配信パターン異常検知
- → 疑わしいアカウントをフラグ

**[第2層：行動スコアリング]**
- ブロック率
- 通報率
- 未読率
- 配信頻度の変動
- → 総合スコアが閾値を超えるとフラグ

**[第3層：人的レビュー]**
- フラグが立ったアカウントを目視確認
- コンテンツ全体の「印象」で判断
- → 最終判断（理由は説明されない）

### 1.4 重要な前提
- 凍結は予告なし：事前警告なく突然停止される
- 理由は説明されない：LINE公式ガイドラインに「説明する義務を負わない」と明記
- 復旧は困難：基本的に取り返しがつかない
- 認証済アカウントでも凍結される：認証は免罪符ではない

### 2.3 安全な言い換えパターン

**基本的な言い換え**
| NG表現 | 安全な言い換え |
| :--- | :--- |
| 月収100万円も可能！ | 売上アップを目指す仕組みをご紹介 |
| 誰でも簡単に稼げる | 未経験からでも取り組める方法 |
| 副業で収入倍増 | 本業以外にも活かせるスキル支援 |
| 投資で資産を増やす | 賢いお金の使い方や管理術 |
| 稼ぐ | 報酬を得る |

**言い換えの原則**
- **【原則1】結果訴求 → プロセス訴求**
    - NG: 「月100万稼げる」
    - OK: 「仕組みの作り方を学ぶ」
- **【原則2】直接訴求 → 間接訴求**
    - NG: 「収入が上がる」
    - OK: 「スキルが身につく→結果として収益に」
- **【原則3】煽り → 価値提供**
    - NG: 「今すぐ登録！」
    - OK: 「〇〇でお困りの方へ」

### 2.4 ピリオド挿入テクニック
どうしても使う必要がある場合の回避策：

**有効な挿入文字：**
- 「.」（ピリオド）→ 効果あり
- 「/」（スラッシュ）→ 効果あり

**無効な挿入文字：**
- 「　」（透明スペース）→ 効果なし
- 「・」（中黒）→ 効果なし

| 元表現 | ピリオド挿入後 |
| :--- | :--- |
| 起業 | 起.業 |
| 副業 | 副.業 |
| 月収100万 | 月.収.100万 |

**注意:** ピリオド挿入は緊急回避策。本質が同じなら依然としてリスクあり。

## 3. 行動リスク：どう送るか

### 3.1 絶対NG行動
| 行動 | 問題点 | 結果 |
| :--- | :--- | :--- |
| 1日に複数回メール配信 | スパムパターン | 到達率低下 |
| 急に大量配信（2倍以上） | Gmailでも即ブロック | 不達 |
| 短期間に同一URL連投 | スパム判定 | 凍結リスク |
| セールスレター直送 | BAN率上昇 | 凍結リスク |
| 全リストに一斉送信 | 通報率上昇 | 凍結リスク |
| 他社システムから一気にリスト移行 | レピュテーションゼロ | スパム判定 |

### 3.2 安全な行動パターン
| 推奨行動 | 理由 | 具体的方法 |
| :--- | :--- | :--- |
| 定期配信を維持 | 急な増加を避ける | 週1-2回の定期配信を習慣化 |
| 徐々に送信量を増やす | レピュテーション構築 | 後述の移行スケジュール参照 |
| クッションページを挟む | セールスレター直送を避ける | LP→詳細ページ→販売ページ |
| エンゲージ高い人にのみ配信 | 通報率低下 | クリック者・タップ者に絞る |
| 開封しない人には送らない | リスト品質維持 | 定期的に非アクティブを解除 |
| ラベル/タグで配信制御 | 反応者のみに絞る | ボタンタップ者のみに配信 |
| 遅延配信を活用 | 一度に大量送信を避ける | 時間分散で配信 |

### 3.3 LINE BAN対策：運用レベル
**【基本方針】**
- 興味のある方だけに友だち追加させる
    - 通報するような方には訴求しない
- 全リストに一斉送信はしない
    - 一斉送信が必要な場合はエンゲージ高い方だけに送る
- ステップ配信もエンゲージが高い方だけに送る
- 際どい単語は使わない
- 独自ドメインベースでLINE公式アカウントと連携する
    - 影響は軽微だが、リスク分散になる

## 4. リスト品質リスク：誰に送るか

### 4.1 事故事例（実際に発生したケース）
| 事故パターン | 結果 |
| :--- | :--- |
| エラーアドレスが含まれているリストをインポート | 配信停止 |
| 他者から購入したメールアドレス、共有されたメールアドレスが含まれているリストをインポート | スパム判定 |
| 解除済みのメールアドレスが含まれているリストをインポート | 法的リスク＋凍結 |
| 取得元が不明、登録元が不明なメールアドレスが含まれているリストをインポート | スパム判定 |
| 数ヶ月以上メール配信を行っていないメールアドレスが含まれているリストをインポート | 忘れられている→通報 |
| 遵守すべき事項を知らずに配信して、スパムとして判断される | 凍結 |
| 他の配信システムから一度に全てのリストをインポート | 即スパム判定 |

### 4.2 リスト移行の正しい手順
**他社システムからの移行スケジュール（UTAGE推奨）**

| 日数 | 送信件数 |
| :--- | :--- |
| 1日目 | 100件インポートして送信 |
| 2日目 | 残りのリストのうち100〜300件インポートして送信 |
| 3日目 | 残りのリストのうち100〜500件インポートして送信 |
| 4日目 | 残りのリストのうち100〜800件インポートして送信 |
| 5日目 | 残りのリストのうち100〜1000件インポートして送信 |
| 6日目 | 残りのリストのうち100〜1300件インポートして送信 |
| 7日目 | 残りのリストのうち100〜1600件インポートして送信 |
| 8日目 | 残りのリストのうち100〜1900件インポートして送信 |

**重要ポイント:**
- 配信エラーになっていないか毎日確認する
- 配信不可のアドレスが絶対に含まれていないことを確認
- 配信実績を積むまでは1日1通までしか配信しない
- DKIM、DMARC認証を事前に設定しておく
- 作業ミスすると届かなくなるので超入念に作業を実施

### 4.3 リストクリーニングの原則
**【必須事項】**
- エラーアドレスの除去（バウンスしたアドレスは即削除）
- 解除済みアドレスの除去（法的にも必須）
- 長期間未開封者の定期的な解除
- オプトインした方のみに配信（登録を忘れている方には送信しない）

**【推奨事項】**
- 開封していない方の定期的な解除
- 反応がある人にのみ配信を続ける
- 忘れられないために定期的にコミュニケーションしておく

## 5. 構造リスク：事業の組み方

### 5.1 大前提：BANは起きる前提で設計する
**【マインドセット】**
- BANは「もし起きたら」ではなく「いつ起きるか」
- 単一チャネル依存は事業リスク
- 復旧より「影響最小化」「切り替え速度」を重視
- 予備の準備コストは保険料と考える

### 5.2 BANされた際の影響を小さくする対策
| 対策 | 詳細 |
| :--- | :--- |
| BANされることを前提にフローを組んでおく | 最初から想定した設計 |
| セールスまでのシナリオを細分化して複数のLINEアカウントを跨ぐようにし全てのLINEアカウントが落ちないようにする | 用途別アカウント分離 |
| 外部からの登録用LINE、セミナー・個別相談用LINE、購入者用LINEなど | 役割ごとにアカウントを分ける |
| メール、SMSなど別媒体で配信できる体制を用意する | 複数チャネル確保 |
| 自動ウェビナー、セミナー、個別相談では携帯番号を取ってSMSを送れるようにする | SMS送信ルート確保 |
| 広告を出す際などで複数のLINE公式アカウントを用意しリスクを分散させておく | リスク分散 |

### 5.3 BANされた際の復旧を早くする対策
| 対策 | 詳細 |
| :--- | :--- |
| BANされた際に切り替えがすぐにできる対応フローを用意しておく | 手順の事前文書化 |
| BANされた際にUTAGEの配信アカウントコピーの機能を利用する | ワンクリックで設定移行 |
| コピー時に内部の文字列の差し替えが簡単にできるよう独自置き換え文字の機能を利用する | 差し替え簡易化 |
| BANされる前にBANされた後に切り替えるLINEアカウントを用意しておく | 予備アカウント準備 |

### 5.4 絶対に注意すべきポイント
⚠️ **BANされた際に連携情報を差し替えるのはNG**
必ずアカウントコピーを利用する（UTAGE等の場合）

### 5.5 複数媒体の活用
| メイン媒体 | 併用すべきサブ媒体 |
| :--- | :--- |
| メールメイン | LINE + SMS |
| LINEメイン | メール + SMS |
| LINE公式アカウント | LINEオープンチャットも検討 |
| 購入者向け連絡 | Chatwork、Slack、Discordなどのチャットツール |

## 6. 複合リスク：組み合わせで致命傷になるパターン

### 6.1 コンテンツ × 行動 の複合リスク
**[禁止ワード使用] + [大量配信] = 確実に凍結**

例：
- 「稼げる」メッセージを全リストに一斉配信 → 高確率で凍結
- 「月収100万」を1日に複数回配信 → 確実に凍結
- 「副業」+「簡単」+「セミナー誘導」の3点セット → 致命的

**対策の原則:**
- コンテンツで攻めるなら、配信は絞る
- 全リスト配信するなら、表現は控えめに
- 両方攻めるのは自殺行為

### 6.2 リスト品質 × 行動 の複合リスク
**[質の低いリスト] + [大量配信] = 即死**

例：
- 古いリスト + 一斉配信 → 忘れている→通報→凍結
- 無効アドレス多数 + 大量配信 → バウンス率上昇→スパム判定
- 他システム移行 + 一気にインポート → レピュテーションゼロ→即ブロック

**対策:**
- リスト移行は「8日かけて段階的に」
- 無効アドレスは「絶対に」含めない
- 開封しない人は定期的に解除

### 6.3 メール × LINE の相互関係
**[メール到達率低下] ←相関→ [LINE凍結リスク]**

理由：
- 同じ「スパム判定ロジック」が働いている
- 「ユーザーが不快」という本質は同じ
- 片方で問題が出たら、もう片方も見直すべき

## 7. メールサービス別対策

### 7.1 Gmail
- **特性:** 送信通数の急増に敏感
- **基準:** 迷惑メール通報 0.30%未満（結構厳しい）
- **必須対策:** DKIM認証、DMARC認証を通す
- **運用対策:** 送信量を徐々に増やす、少量の送信から始める
- **注意:** 大量の送信をしたことがない場合は突然のボリューム急増を避ける
- **Gmailガイドラインの重要事項:**
    - 自分で意思を持って登録した方以外に配信をしない
    - 送信の許諾を取ることが必須（受信者自身でメールアドレスを登録させる）
    - 送信の許諾を取った記録が必要（いつどこで登録したか）
    - 参照: [Email sender guidelines](https://support.google.com/a/answer/81126)

### 7.2 Yahoo!メール
- **特性:** 文面がチェックされる
- **対策:** スパム業者が使っていそうな表現を避ける
- **注意:** 文面でブロックされやすい

### 7.3 Hotmail
- **特性:** たまに判定がおかしい
- **対策:** 特になし（運の要素）
- **注意:** 誤判定があり得る

### 7.4 iCloud
- **特性:** ポリシーに則っていると思われる場合でも届かないことがある
- **対策:** Apple側の問題なので限界あり
- **注意:** 諦めも必要

### 7.5 携帯キャリアメール
- **特性:** 基本的に届きにくい
- **対策:** SMS併用を検討
- **注意:** メールだけに頼らない

### 7.6 メールが届かない原因まとめ
**【届かない原因】**
- ガイドライン、ポリシーに反してブロックしている
- 大量に送って到着が遅延している
- 実は届いていて迷惑メールに入っている
    - 不着と迷惑メールは違うので注意

**【開封率が下がる要因】**
- メールが届いていない
- 迷惑メールに入っている
- 埋もれている
