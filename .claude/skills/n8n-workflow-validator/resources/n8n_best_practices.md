# n8n ワークフロー ベストプラクティス

## 🏗️ 構造設計

### ノード命名規則
- **名前は「何をするか」が分かるものにする**
  - ❌ `HTTP Request`, `Set`, `IF`
  - ✅ `記事データ取得`, `文字数カウント`, `3000字以上か判定`
- 日本語OK。チーム内で統一する

### ワークフローの分割
- **1ワークフロー = 1つの明確な目的**
- ノード数が20を超えたら分割を検討
- サブワークフローを活用する

---

## ⚠️ エラーハンドリング

### 必須パターン
1. **Error Triggerノード**をワークフローに追加する
2. エラー発生時に**通知** (Slack / Email / LINE) を送る
3. リトライが必要な処理には**Retry on Fail**を設定する

### API呼び出しの注意
- レート制限を考慮する（Wait / Batching）
- タイムアウトを適切に設定する
- 大量データ処理は**Split In Batches**を使う

---

## 🔒 セキュリティ

### Credential管理
- **絶対にAPIキーをノードのパラメータに直接書かない**
- 必ずn8nのCredential機能を使う
- 本番用と開発用でCredentialを分ける

### Webhook
- **認証を必ず設定する**（Header Auth / Basic Auth）
- 不要になったWebhookはすぐ無効化する
- レスポンスに機密情報を含めない

---

## 🚀 パフォーマンス

### メモリ使用量
- 大きなデータを扱う場合は**ストリーミング**を活用
- 不要なフィールドは**Set / Remove Fields**で削除する

### 実行速度
- 並列実行可能な処理は**Split In Batches**で並列化
- 外部API呼び出しは**最小限**に（キャッシュの活用）
- 不要なノードは無効化ではなく**削除**する

---

## 📝 メンテナンス

### コメント・説明
- Sticky Notes でワークフローの概要を記載する
- 複雑なロジックにはノードの Description を記入する

### バージョン管理
- ワークフローのJSONをGitで管理する
- 変更前にエクスポートしてバックアップを取る
- ファイル名に日付を入れる（例: `X自動投稿_20260210.json`）
