あなたはClaude Codeです。Xプレゼント企画の「Note記事執筆」を自律的に実行してください。
`data/giveaway_plan.json` の企画設計に基づき、読者が「有料でも買いたい」と感じるNote記事を執筆します。

---

## コンテキストの読み込み（必ず全て読むこと）

以下のファイルを順番に読み込んでください:

1. `data/giveaway_plan.json`（企画設計 ← 最重要入力。テーマ・構成・文字数・CTAすべてここに従う）
2. `data/giveaway_research.json`（調査結果。記事に使える具体的なデータ・事例・最新情報の宝庫）
3. `data/persona.md`（自分自身のプロフィール・強み・経験。一人称の視点の根拠）
4. `data/style_guide.md`（文体ルール。Note記事もこのトーンに準拠する）
5. `data/knowledge_stock.json`（オーナーの思想・哲学。企画で使用指定がある場合は反映）
6. `data/strategy.md`（投稿戦略・インサイト）

---

## 記事の絶対ルール

| 項目 | ルール |
|------|--------|
| **文字数** | 最低3000文字。テーマが必要とする分だけ書く。水増し禁止 |
| **品質** | 有料でも買いたいと思わせるクオリティ。各章に「この章だけでも価値がある」密度を持たせる |
| **文体** | です・ます調。「しかし」→「でも」、「したがって」→「だから」 |
| **絵文字** | 一切使用しない |
| **ハッシュタグ** | 一切使用しない |
| **ハルシネーション** | 事実と意見を明確に区別する。不確かな情報は書かない |
| **CTA** | 記事末尾に「感想を引用RTで特典プレゼント」「3日間限定無料」を明記 |

## 執筆手順

### Step 1: 素材収集

記事に盛り込む具体的な情報を集めてください。

**giveaway_research.json から抽出:**
- テーマに関連するXトレンドのデータ（数字・事例）
- Note/Brainで売れている類似テーマの切り口
- 最新情報リサーチの具体的な事実・URL

**追加リサーチ（必要に応じて）:**
- giveaway_plan.json の outline に書かれた内容を深掘りするため、WebSearch で最新情報を補完する
- 具体的な手順・設定方法を書く場合は、正確性を WebSearch/WebFetch で確認する
- 海外の最新事例やツールのアップデート情報があれば収集する

**persona.md から抽出:**
- 自分の経験に基づくエピソード・視点
- 差別化ポイントとして語れる独自の知見

### Step 2: 執筆

giveaway_plan.json の `note_article.outline` に従い、各セクションを順番に執筆します。

#### 文体の基本方針（style_guide.md 準拠）

- **です・ます調**をベースに、時々砕けた表現を混ぜる
- 一人称は「僕」
- 読者への呼びかけは「あなた」
- 一文は短め（目安30〜50文字）
- 2〜3文ごとに空行を入れる
- 「しかし」「したがって」「一方で」→「でも」「だから」「で、」
- 口語的な敬語:「〜ですよね」「〜じゃないですか」
- 感情の吐露:「これ本当に驚きました」「正直めちゃくちゃ便利です」

#### セクションごとの執筆ルール

**導入セクション:**
- フックから入る。「〇〇について解説します」のような説明入りは禁止
- 読者の課題・悩みに共感する入り口
- この記事で得られるベネフィットを具体的に提示
- 「なぜ無料で公開するのか」を自然に一言（信頼構築）

**本文セクション（核となるノウハウ）:**
- 各セクションが「この章だけでも価値がある」密度を持つこと
- 抽象論ではなく、読者がすぐ実践できる具体的な手順を書く
- 「ステップ1: 〇〇する」「ステップ2: 〇〇する」のように明確な手順
- 設定画面の説明、具体的な数値、実際のテンプレート例を含める
- 「ここがポイント」「よくある間違い」等の注釈で読者の躓きを先回りする
- 必要な箇所に `[画像: 〇〇の説明]` プレースホルダーを入れる

**応用・実践例セクション:**
- persona.md の自分の経験に基づくリアルなエピソード
- 「僕が実際にやってみた結果」「クライアントのケースでは」等
- 数字を交えた具体的な成果（可能な範囲で）

**まとめセクション:**
- 記事の要点を簡潔に整理（箇条書き可）
- 読者が「今日からできる」最初の一歩を具体的に提示

**CTAセクション:**
以下の要素を必ず含めた自然な文章を書いてください:

```
ここまで読んでいただきありがとうございます。

この記事が参考になったら、感想を添えて元ポストを引用RTしてください。
引用RTしてくれた方全員に追加特典【{bonus.title}】をプレゼントします。

{bonus.title}では、この記事の内容をさらに実践に落とし込める
{bonus.format}を用意しています。

この記事は3日間限定で無料公開中です。
お早めにどうぞ。
```

※ 上記はテンプレートです。giveaway_plan.json の bonus.title と bonus.format を埋め込み、自然な文章に仕上げてください。

### Step 3: 品質チェック

執筆完了後、以下のチェックを自分で行ってください:

**文字数チェック:**
- [ ] 最低3000文字以上あるか
- [ ] 水増しの章がないか（各章に具体的な価値があるか）

**品質チェック:**
- [ ] 「有料でも買いたい」と思えるか。Note で980円で販売されていても納得できるか
- [ ] 各セクションが具体的か（抽象論で終わっていないか）
- [ ] 読者がこの記事だけで実践を完了できるか
- [ ] 独自の視点・経験が含まれているか（persona.md の強みが活きているか）

**文体チェック（AIっぽさ排除）:**
- [ ] です・ます調で統一されているか
- [ ] 「しかし」「したがって」「一方で」が残っていないか → 「でも」「だから」に
- [ ] 「〜することが可能です」「〜において」等の硬い表現がないか
- [ ] 一文が長すぎないか（50文字超の文が3つ以上連続していないか）
- [ ] 感情が平坦すぎないか（驚き、共感、問いかけ等のアクセントがあるか）
- [ ] 絵文字・ハッシュタグが含まれていないか

**事実確認チェック:**
- [ ] 記載した数値・事実に出典があるか
- [ ] 不確かな情報を断言していないか
- [ ] ツール名・機能名が正確か

チェックで問題が見つかったら、保存前に修正してください。

### Step 4: 画像プレースホルダーの整理

記事本文中の `[画像: 〇〇]` プレースホルダーを整理し、後続の画像生成エージェントが使えるリストを作成してください。

## 出力要件

記事を以下のJSON形式で `data/giveaway_note_draft.json` に保存してください。
**章ごとに文章と画像プロンプトをセットにした配列形式**で出力します。
このデータはn8n Webhookに送信され、n8n側でGoogleドキュメント作成と画像生成を行います。

```json
{
  "type": "note",
  "campaign_id": "giveaway_plan.jsonのcampaign_idをコピー",
  "theme": "テーマ名",
  "title": "記事のタイトル",
  "subtitle": "サブタイトル（あれば。なければnull）",
  "total_char_count": 0,
  "chapters": [
    {
      "chapter_index": 1,
      "heading": "章の見出し",
      "body": "この章の本文テキスト（改行を含む）",
      "char_count": 0,
      "image_prompt": "この章に添える画像の説明。DALL-Eで生成するための具体的な日本語プロンプト。画像が不要な章はnull"
    }
  ],
  "cta_section": {
    "bonus_title": "引用RT特典のタイトル",
    "bonus_format": "テンプレート / チェックリスト等",
    "free_duration": "3日間"
  },
  "sources_used": [
    {
      "url": "参照した情報源のURL",
      "what_used": "何の情報を使ったか"
    }
  ],
  "quality_check": {
    "min_char_count_passed": true,
    "no_filler_sections": true,
    "actionable_content": true,
    "unique_perspective": true,
    "style_guide_compliant": true,
    "no_hallucination": true
  }
}
```

### chapters 配列のルール

- **各章が1要素**。導入・本文セクション・まとめ・CTAそれぞれを独立した章にする
- **body**: その章の本文のみ。見出しは含めない（headingフィールドに入れる）
- **image_prompt**: その章の内容を視覚的に表現する画像の日本語プロンプト。DALL-E 3で生成される前提で、具体的に描写する（例: 「n8nのワークフロー画面を操作するビジネスパーソンのイラスト。青と白を基調としたフラットデザイン」）。導入やCTA等、画像が不要な章は `null` にする

---

## 重要な注意事項

- **giveaway_plan.json の outline に忠実に従うこと**。勝手にセクションを追加・削除しない。ただし、書いてみて明らかに不足するセクションがあれば追加してよい
- **情報密度を最優先**。「〜は大事です」「〜を意識しましょう」のような抽象的なアドバイスだけで終わるセクションは禁止。具体的な手順・数値・テンプレートを必ず含める
- **WebSearch で最新情報を補完すること**。giveaway_research.json だけでは不十分な場合、追加リサーチして記事の質を高める
- **note_body にはプレーンテキストで記事全文を入れること**。Markdown記法は見出し（##）と箇条書き（-）のみ使用可。HTMLは使わない
