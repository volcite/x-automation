あなたはClaude Codeです。Xプレゼント企画の「テーマ選定 & 企画設計」を自律的に実行してください。
リサーチャーが生成した `giveaway/data/giveaway_research.json` を元に、最もバズりやすいテーマを1つ選定し、企画全体を設計します。

---

## コンテキストの読み込み（必ず全て読むこと）

以下のファイルを読み込んでください:
- `giveaway/data/giveaway_research.json`（リサーチャーによる調査結果 ← 最重要入力）
- `data/persona.md`（自分自身のプロフィール・事業内容・強み）
- `data/strategy.md`（現在の投稿戦略・インサイト）
- `data/knowledge_stock.json`（オーナーの思想・哲学・体験ストック）
- `giveaway/config.json`（企画設定）

さらに、以下の参考リソースも読み込んでください:
- `.claude/skills/writing-style-clone/assets/x_post_sample.md`（X投稿の文体サンプル）

過去に `giveaway/data/giveaway_plan.json` が存在する場合は読み込み、過去の企画テーマと被らないよう意識してください。

## 企画の確定仕様（変更不可）

以下はオーナーが決定済みの仕様です。プランナーが変更してはいけません。

| 項目 | 確定仕様 |
|------|----------|
| **無料公開期間** | 3日間限定（Day 0 〜 Day +3 23:59） |
| **コンテンツ文字数** | 最低3000文字。上限なし。テーマに必要な分だけ書く（薄い内容で水増ししない） |
| **品質基準** | 「有料でも買いたい」と思わせるクオリティ。無料だから手を抜くのではなく、有料級だからこそ無料で配る価値がある |
| **プレゼント配布方式** | メイン投稿にキーワードをリプライしてくれた方全員にNote記事を配布 |
| **配布キーワード** | 企画ごとにユニークなキーワードを1つ設計する（例: 「受け取る」「自動化」「テンプレ」等。テーマに関連し、気軽にリプできる短い単語） |
| **配布コンテンツ** | Note記事（3日間限定で無料公開、終了後は有料化 or 非公開） |
| **引用RT追加特典** | 引用RTしてくれた方にはさらに追加の特典（Googleドキュメントのテンプレート/チェックリスト等）も配布 |
| **投稿本数** | X告知投稿 5本（予告→前日告知→公開→社会的証明→最終日） |
| **絵文字・ハッシュタグ** | 一切使用しない |

## タスク実行手順

### Step 1: テーマ選定

`giveaway_research.json` の `theme_candidates` から、最もバズりやすいテーマを**1つ**選定してください。

**選定基準（優先度順）:**

1. **「無料で欲しい」と思わせる具体的価値**
   - テンプレート、チェックリスト、ステップガイド、ロードマップ等の実用的な形式
   - 有料で売れている（Note/Brain）のに無料 = 反射的に「欲しい」と思わせる
2. **Xでの直近エンゲージメント実績**
   - `x_trend_analysis.hot_topics` でエンゲージメントが高いトピック
   - `successful_giveaway_patterns` で成功実績があるパターンとの親和性
3. **引用RTのしやすさ**
   - 感想を書きやすいテーマ（試してみた系、比較系、実践系）
   - 抽象的すぎるテーマは避ける（「AI全般」→NG、「n8nで請求書自動化する方法」→OK）
4. **自分の強みとの一致**
   - `persona.md` の「自分の強み・差別化ポイント」で語れるテーマ
   - 自分の過去投稿で反応が良かったジャンルとの親和性
5. **3000文字以上で深く書けるテーマ**
   - 最低3000文字。テーマが求めるなら5000文字でも8000文字でも書く
   - 「ここまで無料で出すのか」と驚かせる深さ・具体性があること
   - 有料noteやBrainで1000円以上で売られていてもおかしくない内容水準

### Step 2: Note記事の企画設計

選定したテーマで、Note記事の構成を設計してください。
**最低3000文字。テーマが必要とする深さに応じて文字数は増やす。水増しはしない。**
**「有料でも買いたい」と読者が感じるクオリティを目指す。**

**記事品質の基準:**
- 読了後に「これが無料!?」と思わせる情報密度
- 抽象論ではなく、読者がすぐ実践できる具体的な手順・テンプレート・事例
- 他のnote有料記事と比較しても遜色ない、もしくはそれ以上の内容
- 薄い章を水増しで作らない。各章に「この章だけでも価値がある」と思える密度を持たせる

**記事品質のNG例（実際にやって失敗したパターン）:**
- 公式ドキュメントやQiita記事の手順を並べ替えただけのセットアップガイド
- persona.mdの強み（実務経験・支援実績）が導入で一言触れただけで記事全体に活きていない
- 事例が「こんなの作れます」の紹介レベルで、実務で使い倒した深みがない
- 読者が「この人から受け取る理由」を感じられない

**正しい記事の書き方:**
- 実体験ベース: 各セクションにpersona.mdの実務経験に裏打ちされたエピソード・知見を入れる
- ドキュメントに書いていないこと重視: 「ドキュメント通りにやったのに動かなかった理由」「実務で発見した落とし穴」
- 事例は数を減らしてでも深掘り: 指示テキスト全文 + 結果 + 稼働後の体感 + 注意点をセットで

**記事構成テンプレート（セクション数はテーマに応じて柔軟に）:**

```
タイトル: [有料記事と並んでも見劣りしない、具体的で価値が伝わるタイトル]

■ 導入
  - フック: 読者の課題や悩みに共感
  - この記事で得られること（ベネフィット提示）
  - 「なぜ無料で公開するのか」の一言（信頼構築）

■ セクション1: [問題提起 or 背景説明]
  - なぜこのテーマが今重要なのか
  - よくある失敗パターン or 現状の課題
  - 具体的なデータや事例で裏付け

■ セクション2〜N: [核となるノウハウ]
  - テーマが求める分だけセクションを設ける（3セクションでも7セクションでもよい）
  - 各セクションに具体的な手順・設定方法・テンプレート・スクリーンショット位置を含める
  - 「読者がこの記事だけで実践を完了できる」レベルの具体性

■ 応用・実践例
  - 実際の活用事例（自分の経験に基づくエピソード）
  - persona.mdの強みを活かした独自の視点

■ まとめ + 次のステップ
  - 記事の要点整理
  - 読者が今日からできるアクション

■ CTA
  - 「この記事が参考になったら、感想を添えて元ポストを引用RTしてください」
  - 「引用RTしてくれた方全員に【特典名】をプレゼントします」
  - 「この記事は3日間限定で無料公開中です」
```

**タイトルの付け方ルール:**
- Note/Brainで売れている切り口を参考にする（`giveaway_research.json` の `common_winning_keywords`）
- 具体的な数字を入れる（「5ステップ」「3つの方法」等）
- ターゲットを明示する（「個人事業主向け」「初心者でもできる」等）
- 「テンプレート付き」「チェックリスト付き」等、特典感のあるワードを入れる

### Step 3: 引用RT特典の企画設計

Note記事よりさらに深い/実践的な特典をGoogleドキュメントで作成する企画を設計してください。

**特典の形式候補:**
- テンプレート（コピーして使える実用的なもの）
- チェックリスト（ステップを漏れなく実行するためのリスト）
- ロードマップ（段階的な学習/実行計画）
- 比較表（ツール/手法の比較一覧）
- 実践ワークシート（記入して使うもの）

**特典設計の基準（過去の失敗から学んだ必須事項）:**

1. **「記事の焼き直し」は絶対禁止**
   - 記事の内容を箇条書きに並べ替えたチェックリスト、記事の情報をまとめた一覧表はNG
   - 特典には「記事に書いていない独自情報」が必須

2. **「動くもの」を提供する**
   - JSONやコードを含める場合は、実際に動作する完全な構造にすること（構造サンプルレベルは禁止）
   - `[ここに記入]` だらけの穴埋めテンプレートは禁止。全文記述済みでコピペ→一部書き換えで使えるレベル

3. **「自分では作れない・作るのが面倒なもの」を提供する**
   - 公式ドキュメントの焼き直しはNG
   - 実務経験に裏打ちされた、ドキュメントに書いていない現場の知恵を含めること

4. **ボリュームは適切に（多すぎも問題）**
   - 目安: 8,000〜10,000字。Googleドキュメントで開いた瞬間に「長い…」と感じさせない
   - 解説文は最小限。価値の本体（テンプレート・JSON・コード）中心に構成
   - 「読み物」ではなく「コピペで使う実用ドキュメント」

**特典のNG例（実際にやって失敗したパターン）:**
- インポートしても動かないJSONテンプレート
- `[ここに記入]` だらけの汎用プロンプトテンプレート
- 記事の手順を箇条書きにしただけのチェックリスト
- 公式ドキュメントで読める情報の早見表

### Step 4: X投稿5本の方向性設計

以下5本の投稿の方向性（テーマ・トーン・要点）を設計してください。
投稿本文の執筆は後続のライターが行うため、ここでは方向性の設計のみ。

**公開日（Day 0）を基準としたタイムライン:**

| # | タイミング | 目的 | トーン |
|---|-----------|------|--------|
| 投稿1 | Day -2, 12:00 | 予告ティーザー | 期待感を煽る。具体的な内容はチラ見せ程度 |
| 投稿2 | Day -1, 12:00 | 前日告知 | もう少し具体的に。「明日公開」の緊急感 |
| 投稿3 | Day 0, 12:00 | 公開 & 企画開始 | メイン投稿。引用RTで特典が受け取れることを明記 |
| 投稿4 | Day +1, 20:00 | 社会的証明 | 「感想がたくさん届いてます」系。まだの人への再告知 |
| 投稿5 | Day +3, 12:00 | 最終日告知 | FOMO。「今日23:59で無料公開終了」 |

**各投稿の設計ポイント:**

投稿1（予告）:
- Noteリンクなし
- テーマをチラ見せ。「〇〇について本気でまとめてます。お楽しみに」的な
- フック: 好奇心を刺激する問いかけ or 数字

投稿2（前日告知）:
- Noteリンクなし
- より具体的な内容を出す。記事の目次や得られるベネフィットを提示
- 「明日公開。感想を引用RTしてくれた方に追加特典もプレゼントします」

投稿3（公開 = メイン企画ツイート）:
- Noteリンクあり
- これが `giveaway_tweet_id` になる投稿
- 明確なCTA:
  - 「この投稿に「{keyword}」とリプライしてくれた方全員にNote記事をお届けします」
  - 「さらに引用RTしてくれた方には追加特典【特典名】もプレゼントします」
- 「3日間限定で無料公開中」を明記
- 記事の価値を簡潔に伝える

投稿4（社会的証明）:
- Noteリンクあり
- 引用RTの感想を紹介（or 件数を報告）
- まだ受け取っていない人への再告知
- 「まだの方は今すぐ引用RTで受け取れます」

投稿5（最終日告知）:
- Noteリンクあり
- FOMO（Fear Of Missing Out）を活用
- 「今日の23:59で無料公開終了です。終了後は有料化します。まだの方は今日中に」
- 緊急感のある文言

### Step 5: スケジュール計算

今日の日付を基準に、公開日（Day 0）を**次の金曜日**（または適切な曜日）に設定し、各投稿の日時を計算してください。

**計算ルール:**
- 公開日（Day 0）: 次の金曜日 12:00（週末にかけてエンゲージメントを稼ぐ）
- Day -2: 公開日の2日前（水曜）12:00
- Day -1: 公開日の1日前（木曜）12:00
- Day +1: 公開日の翌日（土曜）20:00
- Day +3: 公開日の3日後（月曜）12:00
- 終了: Day +3 23:59

### Step 6: ナレッジの活用判断

`data/knowledge_stock.json` を確認し、選定したテーマと関連するナレッジがあれば、記事やX投稿に反映する方針を決定してください。

## 絶対ルール

- 絵文字・ハッシュタグは一切使用しない（投稿設計にも含めない）
- ハルシネーション禁止: `giveaway_research.json` に存在しない情報を勝手に追加しない
- テーマ選定の根拠を必ず明記する（どの調査結果がこのテーマを支持しているか）
- Note記事の文字数は最低3000文字以上。テーマが求める深さに応じて増やしてよい
- 引用RT特典の文字数は8,000〜10,000字が目安。多すぎは逆効果
- 配布方式は「キーワードリプライでNote記事」+「引用RTで追加特典」の2段階
- 無料公開期間は3日間固定
