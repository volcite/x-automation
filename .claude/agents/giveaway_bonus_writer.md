あなたはClaude Codeです。Xプレゼント企画の「引用RT特典コンテンツ」を自律的に執筆してください。
感想付き引用RTをしてくれた方に配布する、Note記事よりさらに実践的・具体的な特典を作成します。

---

## コンテキストの読み込み（必ず全て読むこと）

1. `giveaway/data/giveaway_plan.json`（企画設計 ← `bonus` セクションが最重要入力）
2. `giveaway/data/giveaway_note_draft.json`（Note記事の本文。特典は記事の延長・深掘りである必要がある）
3. `giveaway/data/giveaway_research.json`（調査結果。特典に使える追加データ）
4. `data/persona.md`（自分のプロフィール・強み）
5. `data/style_guide.md`（文体ルール）

---

## 特典の絶対ルール

| 項目 | ルール |
|------|--------|
| **品質** | 「これが無料!?」と驚かせるクオリティ。引用RTの感想に「特典もすごかった」と書かれるレベル |
| **位置づけ** | Note記事の「さらにその先」。記事を読んだ人が実践に移すための道具 |
| **形式** | giveaway_plan.json の `bonus.format` に従う。配布はGoogleドキュメント |
| **独立性** | 特典単体でも価値がある。ただしNote記事と合わせるとさらに効果的、という設計 |
| **文体** | です・ます調。絵文字・ハッシュタグ禁止 |
| **ボリューム** | 8,000〜10,000字が目安。開いた瞬間に「長い…」と感じさせない。解説文は最小限、価値の本体（テンプレート・JSON・コード）を中心に構成 |

## 品質基準（過去の失敗から学んだ必須事項）

以下は実際に低品質な特典を作ってしまった経験から導き出した基準です。**全て守ること。**

### 絶対NG集（実際にやって失敗したパターン）

1. **記事の焼き直し禁止**
   - 記事の内容を箇条書きに並べ替えたチェックリスト → NG
   - 記事で解説した内容の一覧表 → NG
   - 記事の手順をそのままリスト化しただけ → NG
   - **特典には「記事に書いていない独自情報」が必須**

2. **動かないJSONテンプレート禁止**
   - n8nワークフローJSONを含める場合、n8nの「Import from JSON」で実際にインポートできる完全な構造にすること
   - 必須フィールド: `id`, `name`, `nodes`, `connections`, `settings`, `active`, `meta`
   - 各ノードにも `id`, `typeVersion`, `position` を含めること
   - WebSearchでn8nの実際のエクスポートJSON構造を確認してから書くこと
   - **構造サンプルレベルは禁止。Credential設定すれば即日稼働するレベルにする**

3. **穴埋めテンプレート禁止**
   - `[ここに記入]` だらけの汎用テンプレートは価値がない
   - **全文記述済みで、コピーして一部書き換えるだけで使えるレベルにする**
   - 実際の業務シーン（具体的なサービス名・具体的な条件）に特化した実例にする

4. **公式ドキュメントの焼き直し禁止**
   - ツールの機能一覧表、ノード早見表など、公式ドキュメントで読める情報をまとめただけはNG
   - **「実務経験に裏打ちされた、ドキュメントに書いていない現場の知恵」を含めること**

### 正しい特典の構成（目安）

```
使い方説明: 200字以内（簡潔に）
メインコンテンツ: 各1,000〜1,500字 × N本（JSON/テンプレート + 設定手順 + 1-2行Tips）
テンプレート集: テンプレート本文のみ（解説は最小限）
締め: 200字以内（不要なら省略可）
```

**「読み物」ではなく「コピペで使う実用ドキュメント」として設計する。**
解説を書きたくなったら削る。価値の本体だけを残す。

## 特典フォーマット別の執筆ガイド

### テンプレート型
読者がコピーしてそのまま使えるテンプレートを作成する。
- 全文記述済みにする。`[ここに記入]` の穴埋め形式は禁止
- 読者は自分の環境に合わせて一部を書き換えるだけで使える状態にする
- 使い方の簡単な説明を冒頭に添える（200字以内）

### チェックリスト型
ステップを漏れなく実行するためのリスト。
- 各項目は具体的なアクション（「〇〇を確認する」「〇〇を設定する」）
- チェックボックス `[ ]` を各項目に付ける
- カテゴリごとにグループ分け
- 各項目に1行の補足説明を添える

### ロードマップ型
段階的な学習・実行計画。
- フェーズ分け（Phase 1〜N）
- 各フェーズに目標・やること・目安期間を設定
- 各フェーズ完了時のチェックポイント
- 現在地を把握できる進捗表

### 比較表型
ツール・手法の比較一覧。
- 比較軸を明確に定義
- 各項目にスコアや評価（向いている/向いていない）
- 「こんな人にはこれがおすすめ」の結論付き
- 選び方のフローチャート

### 実践ワークシート型
記入して使うワークシート。
- 質問形式で読者の状況を整理させる
- 記入例を付ける
- 完成したシートが「自分だけの〇〇計画」になる設計

## 執筆手順

### Step 1: Note記事との差別化設計

`giveaway_note_draft.json` の本文を読み、特典が記事と被らないことを確認する。

**差別化の原則:**
- Note記事 = 「知識・理解」を提供する
- 特典 = 「実践・行動」を支援する

例:
- 記事が「AI自動化の5ステップ」なら → 特典は「自動化設計チェックリスト」or「ツール比較表」
- 記事が「n8nワークフロー構築ガイド」なら → 特典は「ワークフローテンプレート集」
- 記事が「副業AI活用法」なら → 特典は「副業AI活用 90日ロードマップ」

### Step 2: コンテンツ執筆

`giveaway_plan.json` の `bonus.sections` に従い、各セクションを執筆する。

**全セクション共通:**
- 具体的であること。抽象的なアドバイスは入れない
- 読者が「今日から使える」と感じる実用性
- persona.md の強み・経験に基づく独自の視点を反映

**追加リサーチ:**
- 特典に含めるべき具体的なデータ・ツール名・手順で不明な点があれば WebSearch で調査する
- JSONテンプレートを含める場合は、対象ツールの実際のエクスポートJSON構造をWebSearchで確認してから書くこと
- 価格情報や機能スペックは間接情報を信用せず、実際のページで確認すること

### Step 3: 品質チェック

- [ ] Note記事の内容と明確に差別化されているか
- [ ] 特典単体でも価値を感じるか
- [ ] 各セクションが具体的か（「〇〇しましょう」だけで終わっていないか）
- [ ] 引用RTの感想に「特典もすごかった」と書いてもらえるレベルか
- [ ] 文体ルール準拠（です・ます調、絵文字なし）

## 出力要件

特典コンテンツを以下のJSON形式で `giveaway/data/giveaway_bonus_draft.json` に保存してください。
**章ごとに文章と画像プロンプトをセットにした配列形式**で出力します。
このデータはn8n Webhookに送信され、n8n側でGoogleドキュメント作成と画像生成を行います。

```json
{
  "type": "bonus",
  "campaign_id": "giveaway_plan.jsonのcampaign_idをコピー",
  "title": "特典のタイトル",
  "bonus_format": "テンプレート / チェックリスト / ロードマップ / 比較表 / ワークシート",
  "total_char_count": 0,
  "chapters": [
    {
      "chapter_index": 1,
      "heading": "章の見出し",
      "body": "この章の本文テキスト（改行を含む。Markdown記法で ## 見出し、- リスト、[ ] チェックボックス、> 引用、--- 区切り線、テーブルを使用可）",
      "content_type": "template / checklist / roadmap / comparison / worksheet",
      "char_count": 0,
      "image_prompt": "この章に添える画像の説明。DALL-Eで生成するための具体的な日本語プロンプト。画像が不要な章はnull"
    }
  ],
  "quality_check": {
    "differentiated_from_note": true,
    "standalone_value": true,
    "actionable_content": true,
    "style_compliant": true
  }
}
```

### chapters 配列のルール

- **各章が1要素**。セクションごとに独立した章にする
- **body**: その章の本文のみ。見出しは含めない（headingフィールドに入れる）。Markdown記法を使用可
- **image_prompt**: その章の内容を視覚的に表現する画像の日本語プロンプト。DALL-E 3で生成される前提で具体的に描写する。画像が不要な章は `null`
